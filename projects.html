PhishNet AI

A Chrome extension that catches phishing sites in real time, blocks them automatically, and explains why a page is dangerous in plain English.

Type
Browser extension

Manifest V3, JS
APIs
VirusTotal · Gemini
Team
Hackathon team

# What it does

PhishNet AI sits in the browser and watches where you navigate. When you hit a site it hasn't cleared, it checks the URL against threat intelligence, and if the page looks malicious it blocks the navigation before the page can do any damage, then tells you, in human language, what tipped it off.

  • Scans every URL you visit against VirusTotal's aggregated threat database.
  • Feeds flagged results to Gemini, which writes a short, non-technical explanation of the risk.
  • Automatically blocks access to domains identified as phishing, with a warning interstitial.

# How it's built

Manifest V3Service Workerchrome.tabschrome.storageVirusTotal v3Gemini 2.0 FlashVanilla JS

A background service worker (background.js) listens to chrome.tabs.onUpdated on every navigation and runs a two-stage check. First, the URL is matched against a ~150-entry list of remote-monitoring/RMM tool domains that tech-support scams routinely abuse. If it hits, the tab is redirected to a custom intercept.html warning the user can bypass per-domain (the bypass list lives in chrome.storage.local). Otherwise the URL is base64url-encoded and queried against the VirusTotal v3 API; if last_analysis_stats.malicious > 0, the flagged-vendor list is pulled out of last_analysis_results, handed to Gemini 2.0 Flash with a prompt asking for a 2–3 sentence plain-English explanation, stashed in chrome.storage.local, and the tab is redirected to blocked.html. A small popup.html exposes a master enable/disable toggle.

chrome.tabs.onUpdated  (every navigation)
      │
      ├──► matches RMM-tool list?  ──yes──►  intercept.html  (user can bypass)
      │
      ▼
  VirusTotal v3 API  (URL → base64url → /api/v3/urls/{id})
      │
      ├──► clean → let it through
      │
      ▼ flagged
  Gemini 2.0 Flash  (vendor list → 2–3 sentence summary)
      │
      ▼
  chrome.storage.local + redirect to blocked.html

# Gripes & issues

Six URL-checking APIs before we picked oneresolved

We evaluated around six different URL-reputation APIs before settling on VirusTotal. The constraint wasn't accuracy alone. we needed enough per-vendor heuristic detail to feed Gemini for the plain English summary. Most of them returned a flat verdict and nothing to explain, which would have made the "tell the user why" half of the project impossible.

Pre-trained URL classifiers were too noisyresolved

The first instinct was to run a pretrained URL-classification model locally. Two problems killed it: a high false-positive rate on benign sites, and inference cost that didn't fit the hackathon budget. Dropping it for a VirusTotal lookup let the extension stay light and made the verdicts auditable. In other words, every block points to specific named vendors that flagged the site.

No good way to plug a database into an MV3 extensiondesign tension

We wanted a local cache of already-checked URLs to cut repeat API calls. Manifest Version 3 doesn't make it easy to hook a real database to a service worker, and chrome.storage isn't sized for a proper URL index. We sidestepped it by leaning on VirusTotal's own database as the cache, where every lookup hits a corpus broader than anything we'd have built ourselves.

MV3's API-routing restrictionspain point

Manifest V3 is strict about how a service worker is allowed to talk to external endpoints, and figuring out the right pattern for routing the VirusTotal + Gemini calls through the background script ate a big chunk of the hackathon. Once it clicked, the whole pipeline collapsed into a single chrome.tabs.onUpdated listener that fans out to both APIs and redirects the tab on a hit. However, getting there was easily the most painful research stretch.

PureMark

# What it does

PureMark started as a simple Markdown-to-PDF converter and grew into a full local-first writing environment. It manages projects under ~/Documents/PureMark (each with its own assets and timestamped backups), gives you a real code editor with Vim keybindings, and renders a live, page-by-page PDF preview as you type. When you're happy, one click exports to PDF, DOCX, a self-contained LaTeX .zip, or a Typst .zip.

  • Live PDF preview: debounced re-render, lazy per-page rendering via IntersectionObserver, and scroll position preserved across rebuilds.
  • Math & diagrams: MathLive for equation editing, and visual builders for Mermaid flowcharts and sequence diagrams with full round-trip parsing.
  • Zero-setup typesetting: Pandoc and Tectonic ship inside the app, so users never install LaTeX.
  • Safety nets: autosave every 5s, rolling backups every 30s (capped at 50 per project).
  • Friendly errors: LaTeX's wall of log output is parsed into human-readable messages, with a raw-log toggle for when you need it.

# Architecture

Tauri 2RustReact 18TypeScriptViteAce editorpdf.js / react-pdfMathLiveMermaidPandoc 3.9Tectonic 0.16

It's a Tauri app: a Rust core exposes #[tauri::command] functions (project CRUD, asset handling, backups, and the render/export pipeline) that the React frontend calls over Tauri's IPC. Rather than depending on whatever's on the user's PATH, PureMark bundles Pandoc and Tectonic (a self-contained LaTeX engine) as Tauri externalBin sidecars, pinned and fetched by a build script.

# Gripes & issues

WebKitGTK taints SVG-drawn canvasesresolved

Mermaid diagrams render to SVG in the browser, then get rasterized to PNG through a <canvas> so Pandoc can embed them. On Linux, Tauri uses the WebKitGTK webview, and it taints any canvas drawn from a blob:-URL SVG, so canvas.toBlob() throws SecurityError. The fix: feed the SVG in as a percent-encoded data: URL instead, which bypasses the taint. (There's a sibling workaround in the build script too, WEBKIT_DISABLE_DMABUF_RENDERER=1, to dodge a WebKitGTK rendering bug that blanks the window on some GPUs.)

Pandoc's --pdf-engine won't take a pathpain point

Pandoc's --pdf-engine only accepts an engine name that it looks up on PATH. It refuses an absolute path. But the bundled binary is named tectonic-x86_64-unknown-linux-gnu and lives in the app's resource dir, not on PATH. The workaround: create a "shim" directory containing a tectonic symlink to the real sidecar, then prepend that dir to PATH just for the Pandoc invocation. On Windows, where symlinks need admin/developer mode, it copies the binary instead.

Mermaid v11 collapses the diagram aspect ratioresolved

Mermaid v11 emits SVGs with width="100%" as a presentation hint. When rasterizing, parseFloat reads that as 100px, squashing the diagram and leaving big whitespace bands in the PNG. Fix: read the intrinsic size from the SVG's viewBox first, only falling back to the width/height attributes (and finally a sane default) if it's missing.

Cold-cache LaTeX warmup takes ~30splatform reality

Tectonic downloads its LaTeX package bundle on first use, so the very first PDF build takes about 30 seconds while everything else is instant. PureMark detects a cold cache (is_tectonic_cache_warm) and shows a one-time "warming up" notice so it doesn't look frozen, and routes the cache under the app's data dir. A fully offline bundled .tar is the eventual fix.

LaTeX errors are hostile to normal userspain point

A stray special character or a missing package makes Tectonic fail with a wall of log output that's meaningless to someone who's never touched LaTeX. The Rust side parses the log into friendly errors (missing-package, undefined-control-sequence with the offending line number, generic build-failure), surfaced as readable messages, with a "RAW" toggle for when you actually want the gory detail. It's a real improvement, but mapping every possible LaTeX failure to a friendly message is a bottomless task.

Preview jank: scroll jumps and overlapping rendersresolved

Two preview headaches, both fixed. First, every rebuild swapped the whole PDF and threw you back to the top, so now it captures the scroll ratio before a render and restores it after, unless you scrolled manually in the meantime. Second, fast typing could kick off a new render before the previous one finished; a real time guard combines those, queuing at most one follow-up instead of stacking them.

Single-platform binaries (for now)tech debt

The fetch script and resolver are wired for x86_64-unknown-linux-gnu today. The Rust resolver already handles Windows naming/EXE suffixes and the console-window detach, so cross-platform builds are mostly a matter of fetching the other platforms' Pandoc/Tectonic binaries. Not completely done yet.

# Why it's deceptively complex

On the surface it's "Markdown in, PDF out." Underneath, it's a two way Rust to JS IPC boundary, a bundled-binary toolchain that has to be located and handled differently per OS, a client-side diagram pipeline fighting a webview's security model, a virtualized LaTeX cache, and a live preview that has to feel instant without re-rendering the world on every keystroke. Most of the engineering here is in the seams between those pieces, which is exactly the undocumented edge-case work I enjoy.

Atop Log Parser & Visualizer

A C++ tool that parses mission log files for CPU and RAM performance data and renders it as interactive d3.js charts on a dynamic web frontend.

Type
Parser + visualizer
Parser

Frontend

# What it does

Built around the Linux atop performance monitor's logs, this tool reads raw mission-log output, extracts CPU and RAM time-series data, and presents it as interactive charts so operators can spot performance regressions and resource spikes at a glance.

  • C++ parser reads and interprets mission log files for CPU/RAM performance data.
  • Interactive charts and graphs rendered with d3.js on a dynamic web frontend.
  • Parsing algorithms optimized to handle very large logs efficiently, cutting analysis time.

# How it's built

C++atopd3.js (vendored)HTML / CSS / JS

A C++ parser ingests raw atop dumps, extracts the CPU/RAM time-series, and emits the report as static HTML. The parser is also the page generator. The frontend is plain HTML/CSS with d3.js for the interactive charts, with no React, no build step, and no package manager involvement. d3.js itself is checked into the source tree as a copy of the minified CDN bundle.

atop log file  (huge, raw)
      │
      ▼
  C++ parser  ──►  CPU / RAM time-series  ──►  emits raw HTML
                                                  │
                                                  ▼
                                       static page + vendored d3.js
                                                  │
                                                  ▼
                                       interactive CPU / RAM charts

# Gripes & issues

Python parser couldn't keep upresolved

The first version was in Python. This is quick to write, but on the kind of multi-gigabyte logs atop produces, parse times were unworkable. I rewrote the whole parser in C++ and the runtime increasedtenfold. Same parsing algorithm, and yielded the same output, but the program was no longer bottlenecked by the Python interpreter.

Sandboxed environment, package-approval frictionpain point

Everything we wanted to install needed sign-off, and the only approved visualization library at the start was matplotlib. matplotlib's static plots couldn't surface the patterns operators were actually looking for in a multi-hour log where you needed to zoom, hover, and compare ranges, not just stare at a snapshot.

Getting d3.js in without an installerworkaround

I pivoted to d3.js for the interactive charts, but the package install path conflicted with other approved tooling. The workaround was crude and effective: drop the entire minified d3 CDN bundle directly into the source tree and serve it as a static asset.

No frontend framework alloweddesign tension

React / Vue / anything modern were off the table, which meant the C++ parser had to be the page generator too. String-building HTML files directly instead of producing JSON for a frontend to consume. Not elegant, but it kept the whole tool to a single self-contained binary plus a folder of static files, which was actually a feature in that environment.

MediCheck AI

An AI-driven pre-diagnosis tool: enter your symptoms, history, and lifestyle, and a custom MedLLaMa2 model returns preliminary health insights.

# What it does

MediCheck AI gives users a quick, accessible first pass at understanding their symptoms. Through a straightforward web form, users describe what they're experiencing along with relevant history and lifestyle factors, and a medical-domain language model returns preliminary insights, explicitly framed as a starting point, not a diagnosis. The whole stack runs locally on consumer hardware, so it works in places without reliable internet.

  • Plain HTML form for age, symptoms, duration, severity, history, surgeries, family history, fitness, and substance use.
  • Powered by MedLLaMa2, a medical-domain fine-tune of Meta's LLaMa2, served locally through Ollama.
  • Designed to run on modest hardware (8 GB RAM, ~4-core CPU), and tested on a Raspberry Pi 5, so it's usable in offline / underserved settings.

# How it's built

PythonFlaskOllamaMedLLaMa2HTML / CSS

The frontend is a static HTML form. Flask receives the POST, stitches the answers into a single structured prompt (with an explicit instruction prepended that the response must lead with a "preliminary diagnosis, not professional medical advice" disclaimer), and pipes that prompt into a locally-running MedLLaMa2 via Ollama. The response streams back token-by-token.

HTML form  (age, symptoms, history, lifestyle, …)
      │
      ▼
  Flask  ──►  stitches answers into a single prompt
      │       (+ disclaimer instruction prepended)
      ▼
  Ollama (local)  ──►  MedLLaMa2 inference  (streamed)
      │
      ▼
  plain-English preliminary insight + safety disclaimer

# Gripes & issues

Running a medical LLM at a hackathonresolved

Big models don't run on hackathon hardware. We deliberately picked MedLLaMa2, which is small enough to serve through Ollama on a consumer laptop with 8 GB of RAM and no GPU. The same configuration runs on a Raspberry Pi 5, which was the whole point: usability in places without a fast internet connection, where calling a hosted LLM isn't an option to begin with.

Hallucinations in a medical contextdesign tension

An LLM confidently inventing a diagnosis is genuinely dangerous here. We handle it at the prompt layer: every request appends an explicit instruction asking the model to lead with "this is a preliminary diagnosis, not a substitute for professional medical advice." It's not a real safety guarantee, but it gets the framing in front of every response without us having to police the UI.

The form is staticfuture iteration

Right now the questionnaire is hardcoded, where you fill in everything up front and submit once. The obvious next step is branching follow-ups: if a symptom is listed, ask about its onset; then duration; then location. The model could do meaningfully better with that kind of progressive context, and it's the iteration we ran out of hackathon hours for.

Ryujinx Configurator

A controller-friendly, fullscreen tool for binding gamepads in the Ryujinx Switch emulator, straight from Steam Big Picture Mode, no desktop, no mouse, and no fiddling with Ryujinx's own config window.

Target

# The problem

Picture couch co-op: friends come over with their controllers, you want to play Mario Kart, and now everyone's waiting while you drop to desktop mode, open Ryujinx, and bind each pad by hand. It's slow, it's awkward, and it pulls you out of the living-room experience entirely.

Ryujinx Configurator fixes that. It's a fully controller-navigable interface you can add to Steam as a non-Steam game and launch right from Big Picture / Gaming Mode, bind everyone's controllers in seconds, hit save, and you're playing. It was built and tested primarily on the Steam Deck (and a Nobara Linux PC).

# How it's built

Python 3.8+pygamepygame._sdl2 (controller)PyInstaller --onefileRyujinx Config.json

It's a single pygame app running fullscreen, driven entirely by a small state machine: MENU (pick a player slot) → DETECT (press a button on the controller/keyboard you want to bind) → MAPPING (the list of Switch inputs) → SELECTOR (choose what each maps to). Input comes through SDL's normalized controller layer, with a fallback to raw joystick button indices for pads SDL doesn't recognize.

The actual job is reading and writing Ryujinx's Config.json. The tool backs up the original on first run, writes atomically (temp file + os.replace), and disables saving entirely if the existing config is unreadable, so it can never corrupt a working setup.

controller / keyboard input
            │  (SDL, normalized + raw fallback)
            ▼
   MENU → DETECT → MAPPING → SELECTOR
            │
            ▼
   GUID conversion (SDL ⇄ Ryujinx)
            │
            ▼
   atomic write → Ryujinx Config.json   (backup kept as .bak)

# The GUID rabbit hole

The single hardest part was figuring out how Ryujinx identifies controllers. It does not use the standard, documented SDL controller GUID. Instead it stores the id as new Guid(sdlGuidHex).ToString(), which means .NET's Guid reverses the byte order of the first three fields (a 4-byte, a 2-byte, and a 2-byte chunk) while leaving the rest alone. In other words, a mixed big-endian/little-endian layout within the same value.

There's an extra twist: Ryujinx zeroes the CRC16 that SDL stuffs into bytes 2–3 (a name/info checksum that varies and shouldn't affect device matching). I verified this against a Bluetooth Switch Pro Controller. SDL reported CRC 0x15b7, Ryujinx wrote 0000. The converter reproduces both behaviors:

b = [guid[i:i+2] for i in range(0, 32, 2)]   # 16 bytes
b[2] = b[3] = "00"                            # zero the CRC16
g1 = b[3] + b[2] + b[1] + b[0]                # reversed (4 bytes)
g2 = b[5] + b[4]                              # reversed (2 bytes)
g3 = b[7] + b[6]                              # reversed (2 bytes)
g4 = b[8] + b[9]                              # as-is
g5 = "".join(b[10:16])                        # as-is

# Gripes & issues

Ryujinx's non-standard GUID formatreverse-engineered

This tripped me up for a solid two hours. Why does Ryujinx not just use standard SDL controller GUIDs? I had to reverse-engineer how the GUIDs were being generated (big-endian for part of it, little-endian for another part of the same GUID, plus a zeroed-out CRC16). Imagine using a documented standard... couldn't be Ryujinx, apparently. The tool now converts cleanly between standard SDL GUIDs and Ryujinx's big-little-endian format.

Steam Input virtualizes your controllerplatform gotcha

When you launch through Steam Big Picture, Steam Input hides your physical pad and presents a virtual one with a different GUID and a generic name. Moonlight game streaming does the same thing. So a binding made one way silently won't match if you then launch Ryujinx a different way.

The fix: launch the tool and Ryujinx the same way, and use the built-in Diagnostics overlay (press D) to compare the exact id/name SDL reports against what Ryujinx lists. I built that overlay specifically to debug this class of "Waiting for controller connection..." mismatch.

Multiple identical controllershardware limitation

Two of the same Xbox pad (or several streamed through Moonlight) share a GUID and are told apart only by a connection-order index: (0), (1), etc. That order can change between sessions, so an assignment can end up mapped to a different person's controller next time. That's how the controllers identify themselves, not something this tool can work around.

Steam won't list the non-Steam app on Linuxworkaround documented

Adding a non-Steam game sometimes just doesn't show the executable in the browse list, on both Steam Deck and my Linux PC. Workaround: add any app that does show up, then edit its properties, point Target at the Config Tool executable (quotes required), set Start in to your home dir (no quotes), no launch options.

It's one big monolithic filetech debt

The whole thing lives in a single ~840-line main.py. It really should be modularized so the same controller-friendly UI could drive config for other emulators too. That's a "later" task.

System package conflicts on buildresolved

My system packages clashed with the app's dependencies, so the build broke until I isolated it. The README now walks through a venv + PyInstaller --onefile flow that sidesteps it entirely.
